# Blockers

Last reviewed: 2026-05-26

This file documents why each market is either implemented, guarded, or stubbed. The project rule is: use official APIs, SDKs, documented endpoints, or protocol contracts only. Do not scrape private data, bypass authentication, or enable live trading/copy trading by default.

## Summary

- Polymarket, Kalshi, PredictIt, Manifold, Metaculus, Myriad, Opinion, Gemini, Predict.fun, XO, Betfair, Limitless Exchange, SX Bet, Azuro, Augur, Omen, and Zeitgeist are implemented adapters today.
- All other markets are present in config and GUI through verified-blocked adapters with exact blocker reasons.
- Live trading remains disabled by default for every market.
- Credentials must stay in `.env`, shell environment variables, OS keychain tooling, or external credential files. They must not be stored in `data/config.json`.

## 2026-05-26 Re-Audit Notes

Article 35 re-checked verified-blocked markets against current official pages and docs. No verified-blocked market was promoted in this pass because every candidate still fails at least one project rule: official production API availability, account/entitlement access, market-data licensing, documented order semantics, wallet/contract safety, or offline fixture coverage.

Notable re-audit outcomes:

- `context_v2`: official Context Markets site now states the product is sunset and markets are closed out. Context SDK/research docs still describe read-only and trading APIs, but the sunset product state makes a new adapter unsafe without an explicit successor API and support terms.
- `hyperliquid`: official Hyperliquid docs now document outcome asset IDs and `outcomeMeta`, but `outcomeMeta` is documented as testnet-only. Keep blocked until production outcome metadata, market lifecycle semantics, and wallet/order safeguards are stable and covered by fixtures.
- `thales_market`: official Thales docs describe API endpoints and contract integration, but support requires chain-specific API schemas, AMM/contract handling, collateral/network coverage, and explicit wallet transaction safeguards before even read-only rows can be tested safely.
- `smarkets`: official Smarkets API terms require account verification, application approval, a setup fee, pre-approved markets/limits, and written approval before platform data can be shared or distributed. Keep blocked without user-provided approval.
- `cme_prediction_markets`: official CME pages show market data APIs and separately licensed Event Contract Information, so support remains blocked without CME data entitlements plus a broker/order path.
- Consumer app products such as Robinhood Prediction Markets, DraftKings Predictions, Fanatics Markets, BetMGM, PrizePicks, Underdog, and Probo remain blocked because official public automation APIs and third-party integration terms are not published.

## Implementation Rules For Clearing A Blocker

1. Add or update an adapter using official/documented APIs only.
2. Add offline fixture tests for every endpoint parser and order/dry-run path.
3. Keep paper/dry-run mode as default.
4. Keep live trading disabled unless the user explicitly enables it in local config.
5. Never commit credentials, cookies, tokens, API keys, private URLs, IPs, or user data.
6. Update `README.md`, `data/config.example.json`, and this file when a blocker changes.
